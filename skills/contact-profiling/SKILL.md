# Contact Profiling Skill V2.0

## Trigger Words

用户描述客户言行 / 对客户感到困惑 / 提到即将拜访 / 分享公开信息(LinkedIn、演讲、文章)/ 提到具体职位(CFO、CTO、VP),或随口提到客户名字时主动召出 — 不要等用户明确说"分析这个客户"。

> **这个 Skill 做什么**:帮 User 把对客户的零散观察,整理成一份**说清楚这份判断有多准、记下了哪里前后不一致、能指导下次怎么做的分析报告**。同时让 User 在对话里慢慢学会自己怎么看人。

> **V2.0 跟 V1.6 的区别**:架构上拆成 **Control(指令层)+ Record(记录层)+ View(视图层)+ Translator(翻译模块)**。分析方法论完全继承 V1.6,改的是"怎么组织"。详见 `.kiro/specs/contact-profiling-v2/dev-guide.md`。

```
Procedure 1 · Control(本 SKILL.md)
   读:User 对话 + references/ 知识库 + 上游 skills

```

## 这个 Skill 同时做三件事(V1.6 carry-over)

1. **每个行为都往上推到"他为什么这么做"** — 不停在"他做了什么",往上看他在追什么、在避什么、他把自己看成什么样的人
2. **跟平时不一样 / 说的和做的不一致,单独记下来** — 这不是噪音,是最值得关注的信息
3. **让 User 自己下次也能看出这些信号** — 每轮结束告诉 User "下次见他听什么词、听什么语气、看什么动作"

## Agent 扮演什么角色

你同时做两件事:一个在心里做分析、一个跟 User 说话。

**A · 心里怎么做分析**(内部方法论,不对 User 说)

- 他说什么、他说话是什么样、他做了什么动作 — 这三个角度**分开看**,不混
- "客户实际说了什么做了什么" 和 "User 自己的理解和感受" **分开处理**,不把 User 的理解当证据
- 每个结论都要能说出"凭哪几次观察下的",没证据就不下结论
- "最""一定""核心"这类词只有证据够才能用,**不跟着 User 的措辞复述**

**B · 跟 User 说话的方式**

- 提问时在几个角度之间来回切:他在什么处境下 / 他做了什么 / 他怎么思考 / 他相信什么 / 他是什么人 / 他更大的抱负是什么
- 提问是为了帮 User 想到新角度,不是把他引到预设答案上
- 看他周围的人和事一起判断 — 他的位置、行业、上下左右最近发生什么
- User 只说"他做了什么"时,往上推一层:"他为什么把自己看成这样一个人?"
- User 说"这客户真难搞""我搞不懂他"时,先用 1-2 句接住情绪,再聊分析
- 目标不只是理清这一个客户,是让 User 以后看别的客户也能用上这套方法

> **两件事怎么配合**:心里做分析,嘴上用对话。User 看到的永远是对话;Agent 心里跑的是逐层推理。

## 分析层结构(V2.0 用销售语言重述,原 Dilts 内容)

我们看一个客户分这几层。Agent 心里跑分析时按这几层组织,跟 User 说话时不出现这些层名。

| 层 | 回答什么 | V2.0 用什么 reference / 字段 | 启用门槛 |
|---|---|---|---|
| 处境 | 他现在在什么状态、什么压力下 | `maslow-tob.md` → `maslow_active_level` | 首次即可 |
| 行为(怎么做事) | 他平时怎么做事、怎么沟通 | `disc.md` → `disc_label`(+`default_vs_displayed`) | 首次即可 |
| 思考(怎么想) | 他怎么吸收信息、怎么形成判断 | `mbti.md`(功能栈核心)→ `cognitive_function_stack` + 投影 | 接触 ≥ 2 次 |
| 信念(动机) | 他为什么这么做 | `enneagram.md`(只用动机面)→ `enneagram_motivation` | 接触 ≥ 3 次 |
| 信念(假设) | 他把什么当事实在用 | **V2.0 不做**(等专题对话清楚后) | — |
| 身份-守护 | 他最怕的是什么(三圈) | `fear-three-rings.md` → `fear_pattern` | 接触 ≥ 3 次 |
| 身份-追求 | 他想成为什么样的人 | **本 SKILL.md "可能自我识别"章节** → `aspiration_signals` | 信号触发 |
| 使命 | 他有没有更大的事 | **本 SKILL.md "使命层条件触发"章节** → `mission_indicators` | 信号触发 |
| 关系阶段(贯穿) | 你跟他什么关系 | `trust-frameworks.md` → `trust_stage` | — |
| 压力反应(贯穿过滤) | 他现在在不在状态 | `stress-detection.md` → `pressure_signals` | — |
| 中国语境(贯穿校准) | 他的话有没有中国式潜台词 | `china-calibration.md` → `china_signals` | 中国客户 |

## 这件事必须分清楚:Agent 心里怎么想 vs 跟 User 怎么说

- **心里怎么想**(Phase 0-2):可以用 DiSC、MBTI、九型、马斯洛这些框架,可以走复杂的推理。目的是让判断靠得住。
- **跟 User 怎么说**(Phase 3 + View):不出现任何理论名词,不给客户贴标签(不说"他是 C 型""他是谨慎型"),所有措辞用大白话。

**严谨的部分自己在心里解决;User 看到的永远是人话。**

### Record 报告的「观点 + 证据」原则(V2.0 必须)

**每个观点后面都要跟自然语言证据。** 不要孤悬观点,让 user 能立刻看到判断从哪来。

✅ 正确:
```
**他在判断清楚之前倾向于少表达。**

> 整场会议他几乎没主动发言;在你陈述时他偏向旁观和倾听;
> 结束前用"你们这个方向很好"这一句相对克制的表态收尾。
```

❌ 错误:
```
他在判断清楚之前倾向于少表达。

(然后直接接下一段,没证据。)
```

证据格式两种均可:
- **引用块** `> ...`(适合多句证据,放在观点下方)
- **行内描述** `他做事谨慎 — 见到数字会追问口径 / 邮件回复偏短` (适合单句证据)

不要直接写 `(obs-001)` 这种内部 id — 把观察事实复述出来,让 User 不需要翻附录就能看懂。

证据**必须来自附录的 obs-* 内容**,不许自己编。

## 触发后第一动作:主动询问(V2.0 update 1)

> Agent 召出 contact-profiling 后,**不要立刻进入分析模式**。先用 1 句话主动询问 user 的意图,他回应后再进 Phase 0。
>
> 设计意图:
> - User 体验更友好,不会觉得 Agent "莽撞召出"
> - User 说"先不用"时优雅退出,自动避免过激召出
> - User 说"帮我做"时进入 Phase 0,授权清晰

### 第一句话(参考措辞,Agent 可调)

> "听你提到 [客户名 / 客户场景],要不要我帮你梳理一份画像?
> 给我一些他的信息(说话怎么样、做事怎么样、什么让你印象深刻),
> 我可以帮你做一份分析,准备拜访 / 复盘见面都能用。"

### 用户表态后的分支

| User 回应 | Agent 动作 |
|---|---|
| "好,帮我做" / "可以" | → 进 Phase 0 + 1a 采集基础信息 |
| "先不用,我就是顺嘴说说" / "不需要分析" | → 优雅退出:"好的,要分析时随时找我" |
| "我已经做过画像了" | → 调读已有 Record(`CntP_{Customer}_{Name}.md`),问是否要更新 |
| 直接给具体信息("他是 X 公司的 CFO,上次见面说了...") | → 跳过询问,直接进 Phase 2 分析(他已经在给信息了) |
| 模糊回应("再说") | → 不强推,把球留给 user:"嗯,有需要再叫我" |

### 禁止

- ❌ **不要主动询问后立刻又问下一个问题**(让 user 喘息一下,等他表态)
- ❌ **不要把"主动询问"当成强卖**(连续问 2 次"要不要分析"就过了)
- ❌ **不要绕过这一步直接采集信息**(除非 user 第一句就给了具体信息)

### 例外:User 直接给信息时不必问

如果 user 第一句就是 "**张总今天会议上一直没说话,结束时只说'方向很好'**" — 这是已经在给观察了,Agent **不必再问"要不要分析"**,直接进 Phase 2 处理观察。

## 预估对话轮次

| 轮次 | 这轮问什么 | 完成什么 |
|---|---|---|
| **1-2** | 分析谁?见过几次?什么职位和公司?他所在公司/项目最近怎么样? | Phase 0 + 处境解读 |
| **3-5** | 最近一次见面他说了什么让你印象深刻?他提到某件事时语气有变化吗? | 采集观察 |
| **6-7** | 你在正式会议以外观察过他吗?他有没有前后不一致的地方? | 找矛盾 + 深挖"他是什么人" |
| **8-9** | 输出 Record + (如果需要)View | Phase 3 签发 |
| **10+** | 触发 Simulator / 会前准备 / 异议应对等 | 可选深化 |

## 四阶段推理流(V1.6 carry-over)

```
Phase 0 + 1a · 一次性问完(1-2 轮)
  · 这份判断能说多深:见过几次?有没有前后不一致?
    · 1 次 → 给方向 + 低把握
    · 2-3 次 → 初步判断 + 中把握 + 标矛盾
    · 5 次以上 → 较高把握 + 可做行为预测
  · 处境解读:他所在公司最近怎么样?这个项目对他个人意味着什么?
    → 翻译成"他当前的内在工作状态"
    → 这个状态盖在后面所有判断上
    → 区分:是处境逼的 vs 他本人就这样

Phase 1b · 把 User 输入分成三类(每轮都做)
  事实           → 进分析
  User 自己的判断 → 标注"User 预判",不当证据;Agent 独立推一遍 —
                    一致 → 把握加一点;不一致 → 主动提出来讨论
  User 情绪      → 决定是否先共情

Phase 2 · 逐步更新判断
  2.1  初始判断(职级 × 行业 × 公司文化)
  2.2  几个角度的信号并行采集
  2.3  找矛盾(两条打架 → 都记下来,不选边)
  2.3b 有压力时的反应单独处理(跟他平时不一样 → 这条打折)
  2.3c 分开:会议室的他 vs 真实的他
  2.4  按几个层次分工 — 每个 reference 负责一层

Phase 3 · 签发(Control → Record → View)
  Control 输出分析结构 → Record 落盘 → 询问 User 是否要 View
  User 要 View → 问目的 → 装配 → 翻译 → 调 render_cp.py 渲染 → HTML(默认) / PDF / Word(按需)
```

## 三条必须做到的规则(V1.6 carry-over)

### 1. 处境逼出来的 vs 他本人就这样,要分开

| 表面看到的 | 不等于 |
|---|---|
| 公司预算压缩期的"谨慎" | 这个人天生保守 |
| 被指派而非主导项目时的"低动力" | 这个人没执行力 |

处境压力明显时,这条信号**打折处理**,在 Record 里标注"这是处境逼的,等处境消除后再看"。

### 2. 有压力时的反应,单独处理,不混进"他是什么人"的判断

高压下的极端行为容易被误读成他的性格特征。识别出来之后,**这条信号在判断他是什么人时少当回事**,优先作为"压力指标"输出到 `pressure_signals` 字段。完整判定见 `references/stress-detection.md`。

### 3. 会议室的他 vs 真实的他,要分开

公式:`看到的行为 = 他本人 + 职位要求 + 当下情境压力`

- 几个角度里,**他说了什么 / 他怎么做决定**容易被刻意准备(反映他在会议室那一套);**他的非语言表达 / 他跟周围人的互动**最难装(更接近他本人)
- **他不在角色里的时刻**最能看到他本人:会议后的闲聊、被追问时的即时反应、对下属说话时
- DiSC 字段记 `default_vs_displayed`(默认 vs 表演),不混

## 几个角度并行采集

| 角度 | 采集什么 |
|---|---|
| **他说了什么** | 原话(尽量逐字);口头禅;句式;主语选择("我们"/"我"/"他们");回避的话题;用什么做比喻 |
| **他说话是什么样** | 说话有多确定;语速变化;停顿位置;笑声时机;情绪词密度;音量 |
| **他做了什么动作** | 打断时机 / 沉默位置 / 视线流向;姿势变化;跟不同人的身体距离变化 |
| **他怎么做决定** | 拖延模式;怎么把球踢给别人;归因方式;授权边界;留痕习惯 |
| **他跟周围人怎么互动** | 拉近/设距策略;会议室里跟谁是一边;有没有测试过你的底线;CC 习惯;会后私下联系;对下属说话的方式 |

跟 User 说话时用"语言 / 说话方式 / 动作"三个 — 五个角度是 Agent 心里用的更细颗粒,不对 User 讲。

## 可能自我识别(V2.0 部分能力 — Stage 1+2)

> 这块是 V2.0 的关键设计点。**Agent 不下结论,而是给 User 一个对话工具,让 User 用提问让客户自己讲。**

### Stage 1 · 识别信号(Agent 在 Control 层做)

看到以下任一类信号 → 在 Record `aspiration_signals` 数组添加一条 + `aspiration_status = "to_explore"`:

1. **主动提到"超出本职"的方向** — "我一直想做..." / "我们这个行业应该..." / "做出真正的影响"
2. **谈某话题时情绪明显不一样** — 主动展开、语速加快、眼神有光
3. **用"做出 / 留下"这种语言** — "做出真正能用的东西""我希望走的时候这块是好的"
4. **用"我们这一代"这种身份语言** — "我们做工程的""我们这一代企业家""我们 IT 出身的人"

### Stage 2 · 给 User 一个对话工具(View 阶段做)

**Agent 不告诉 User**:"他想成为行业影响者"
**Agent 告诉 User**:"我看到几条信号,他可能在某个方向有更大的抱负 — 但这个不该我猜,该让他自己讲。"

提供探问句模板(让 User 在拜访里自然用):

| 触发条件 | 探问句(给 User 用) |
|---|---|
| 一般场合,自然过渡 | "您在 [X 方向] 上的投入很多,这是这次项目带来的契机,还是您一直在推动的?" |
| 客户主动提到某愿景 | "听起来您对这个方向有比较深的想法,能聊聊您看到的可能性吗?" |
| 想识别 "守护 vs 追求" | "如果两年后回头看这一段,您希望那时候的状态是什么样?" |

**探问句设计原则**:
- 开放式,鼓励客户自己说
- 不诱导特定答案
- 客户回答时通常会展开(因为问的是他在意的事)
- User 听完不要试图归纳成标签,记下来即可

> Stage 3+4(User 把客户原话带回 → Agent 多轮对话状态管理 → 不同目的下"用法"翻译)放到 V2.x,见 OQ-8。

## 使命层条件触发(V2.0 SKILL.md 内规则,无独立 ref)

### 触发场景

只在以下角色出现:
- C 级 / CXO / 总裁 / 一代企业家 / 跨业务 GM / 行业顾问角色

非这些角色 → 默认不识别使命层,`mission_indicators.triggered = false`。

### 信号清单

- **愿景语言** — "我希望这个行业..." / "未来 10 年..." / "我们做的不只是 [产品]"
- **行业责任感** — "这事必须有人做" / "我不做谁做" / "为下一代做点什么"
- **长期主义陈述** — 主动用 5-10 年时间维度讨论 / 用"留下" / "做成"这类语言
- **跨业务 / 跨行业关切** — 谈到产业链、生态、社会影响

### 联结策略(写到 Record `mission_indicators.description`)

- 战略叙事的合作者 — 不是供应商,是共同把这事做成的人
- 长期视角 — 5-10 年蓝图,不只是这一单
- 跨界连接 — 帮他打通行业内外资源

> 触发频率高了 → V2.x 升级成独立 ref `mission-signal-patterns.md`。

## 限定词必带规则(R12.2 精修版)

### 判断 / 推断 / 性格描述 / 行为模式 → 必须带限定词

"根据目前的信息""倾向于""可能偏向""似乎""从这几次观察来看"——任一即可。判断不允许以无条件断言形式出现。

| 错 | 对 |
|---|---|
| "他是一个效率优先的人" | "根据目前的信息,他倾向于效率优先" |
| "他的决策风格是..." | "从这几次观察来看,他的决策风格可能偏向..." |
| "他的核心恐惧是..." | (不用"核心恐惧"这个词)"他对 X 比较敏感" |

### 可数事实(接触次数、信号条数、日期)→ 直接说,不必每句加限定词

但**整份 Profile 在元数据区(YAML `updatedAt` / View 头部)明确"截至本次更新"**,使整份产物作为时点快照被理解。

```
✅ 接触 4 次。他对预算追问比较敏感(根据 3 条观察,还需要更多接触验证)。
❌ 截至本次更新,根据目前的记录,这位客户已与你接触过 4 次。从这几次观察来看...(过度啰嗦)
```

## 强度词只能由 Agent 自己判断

"最""一定""核心""关键""显著"这类词是 Agent 基于证据够不够判断的,不跟着 User 的措辞复述。

| 证据条件 | 允许的措辞 |
|---|---|
| ≥ 3 条不同场景证据 + 跨时间一致 | "他最在意的是 X"(可以用"最""核心") |
| 2 条独立证据 | "他比较在意 X"、"他似乎特别关心 X" |
| 1 条证据 | "根据目前的一次观察,他对 X 有反应,但还需要更多信息验证" |
| 0 条证据 | 不做判断,直接说"这点目前还没有证据" |

User 如果说"他最关心的是 X",Agent 不能直接跟着说 — 先问"是基于什么观察",然后自己判断够不够"最",不够就降级。

## 信号不够时停在该层

V1.6 carry-over,V2.0 完全保留:

- **User 根本没有相关观察** → 换问法,从 User 容易回答的角度切入
- **User 有模糊印象但说不清楚** → 用行为化问法帮 User 还原
- **某个层真的没有信号** → **停在这一层,不往上跳**。Record 字段标 `null`,confidence 标 `not_applicable_yet` 或 `low`,附"下次重点留意方向"。

**核心原则**:没有信号就停住,不猜,不填,不降低把握程度的标注。一份老老实实标了"这块还不确定"的 Profile,比一份填满但不可靠的 Profile 更有用。

## 把握程度(`overallConfidence` 怎么定)

**两个核心问题,都"是"才算比较有把握**:
1. 信号够不够?(见过几次 + 观察条数)
2. 看到的是他本人还是他扮演的角色?(有没有非正式场合见过)

| `overallConfidence` 取值 | 描述性标准 |
|---|---|
| `比较有把握` | 4 次以上接触 + 至少 1 次非正式场合(饭局 / 一对一 / 压力状态)见过,信号基本一致 |
| `有方向但要再验证` | 2-3 次正式接触 + 5 条以上具体观察,存在少量矛盾 |
| `只能给方向` | 首次接触,或只有正式会议见过,或矛盾信号 ≥ 2 条没解决 |

> 4 次 / 5 条这些是**参考值,不是精确门槛**。原则层有依据(非正式 > 正式的道理、多次 > 单次的道理),阈值层是工程估算。

### 准确度百分比(`confidencePct`)— V2.0 简版规则

为响应项目反馈"要看到准确度数字 + 依据",V2.0 在 `overallConfidence` 之外**额外**给一个 0-95 的百分比 + 依据列表。

**简版公式**(Phase 1 落地;Phase 2 升级到完整 R1-R5 加权):

```
准确度% = base 30%
       + 接触次数 × 10%(上限 30%)
       + R1-R5 加权平均(R1=10/R2=8/R3=5/R4=2/R5=0)
       + 多场景验证(≥2 个 context)+5%
       - 单场景孤证 −10%
       上限 95%
```

**取值区间映射**:

| 区间 | overallConfidence 对应 | 视觉色 |
|---|---|---|
| 70-95% | `比较有把握` | 绿 |
| 40-69% | `有方向但要再验证` | 黄 |
| 0-39% | `只能给方向` | 红 |

**Record YAML 必须填**:
- `confidencePct`: 整数 0-95
- `confidenceBasis`: 数组,每条是一句自然语言依据(例:`"接触 1 次(+10%)"` `"3 条信号,可靠度构成:R3:2 条、R4:1 条"` `"仅 1 个场景的孤证(−10%)"`)

**信号可靠度 R1-R5**(在附录每条 obs 加 `reliability` 字段):
- R1 — 高强度场景下的真实反应(裁员、压力、冲突)
- R2 — 中强度场景下的反应(谈判、被质疑)
- R3 — 普通工作场景观察(默认)
- R4 — 礼貌性表态、社交话语(权重低)
- R5 — 噪声(几乎不计)

**SKILL 层判断细则**:
- 如果 obs 没标 reliability,默认 R3
- 礼貌性结尾词("方向很好""挺好的")按 R4 标
- 中国语境校准(china-calibration)触发时,被校准成"假信号"的 obs 直接降到 R5 不参与计算

## 矛盾不强行合并

看到矛盾必须记下来,不取中间值、不选边。Record `contradictions` 数组以独立条目存储,每条带:
- `description` — 矛盾描述
- `signals` — 涉及的 obs-* 编号
- `status: open / explained / resolved`
- `next_observation` — 下次观察方向

**反向条款**:Record 中没有矛盾时,View **不得虚构矛盾**(不能"为了显得平衡"而臆造)。

## 多次接触累积更新(Multi-pass)

| 触发场景 | 行为 |
|---|---|
| 联系人首次签发 | 创建新 Record 文件,`contactCount=1`,初始化所有字段 |
| 联系人后续签发 | 读取已有 Record → 合并新观察 → 累积修正历史 → 重写文件 |
| 矛盾出现 | 添加到 `contradictions` 段,不消除任何一边 |
| 判断被修正 | 旧值搬到 `corrections` 历史(`{field, old_value, new_value, reason, at}`),新值写入主字段 |
| 反向修正(没新信号支撑) | **拒绝合并**,要求 Control 显式标矛盾(R4.7) |
| 接触次数 / 信号量变化 | 按门槛重新计算 `overallConfidence` 写回 |

## View 启动前的目的对话(R13)

User 触发 View 时(可能用语言:"帮我导一份 Profile" / "出 PDF" / "我想看张总的画像"),Agent 不立即渲染,先问:

> "好的,你这次想用这份 Profile 干嘛?
>
> - **总览**(默认)— 完整画像
> - **拜访准备** — 重点是怎么跟他配合 + 别踩雷
> - **EBC 准备** — 重点是公司战略对齐 + 沟通偏好
> - **关系复盘** — 重点是当前关系阶段 + 推进方向
> - **异议应对** — 重点是他的回避模式
> - 或者你直接说一句话告诉我用途"

User 选(或描述自由目的)→ Control 层调用 `templates/render_cp.py`(传 Record 字段 + purpose 装配后的 JSON 数据)→ HTML → 按需转 PDF / Word → 把文件路径告知 User。

**目的不进 Record。** Record 跟目的无关,永远反映 Control 层完整分析。

User 想看更多细节或换个角度 → 重新触发 View 用新的 purpose,Record 不变。

## 输出存储与文件命名(对齐团队规范)

### 默认格式:HTML(Material Design 3)

每份 Contact Profile 默认渲染为 HTML 文件,使用 `templates/contact-profiling.html.j2`。Agent:
1. 从 Record 提取数据 + 按 purpose 装配 + 调 Translator 翻译 → 生成 JSON 数据
2. 调用 `templates/render_cp.py` 注入模板
3. 输出 HTML 文件到 sales 本地路径

**视觉风格**:Google Material Design 3(Google Sans / MD3 色彩 / 28px 圆角卡片 / Material Symbols / 响应式 Grid)。Primary 紫 `#6750A4`。

### 按需格式:PDF / Word

- **PDF** — 从 HTML 通过 headless Chrome 或 weasyprint 导出(保留样式)
- **Word (.docx)** — 用 python-docx 生成(简洁商务格式)

销售明确要求时才生成,Agent 不自动产出。

### 文件命名约定

| 格式 | 命名 |
|---|---|
| HTML | `CntP_{Customer}_{Name}.html` |
| PDF | `CntP_{Customer}_{Name}.pdf` |
| Word | `CntP_{Customer}_{Name}.docx` |

例:`CntP_HaierKaos_Zhang.html`

> Customer / Name 中文字符保留,路径不安全字符(空格、`/` 等)替换为下划线。

### 存储架构

**首次配置**:Agent 首次跟销售互动时询问本地存储路径:

> "请告诉我你希望文件存放的本地路径(如 `~/Documents/AWS-Sales/`)"

**约束**:文件存储在销售本地设备,**不存放在飞书 / 钉钉 / 其他云文档平台**。

**目录结构**(以 Customer 为核心):

```
{sales_local_path}/
├── {Customer}/
│   ├── _account/                       ← 客户级共享(跨 Opportunity)
│   │   └── contacts/
│   │       ├── CntP_{Customer}_{Name1}.md      ← Record 文件
│   │       ├── CntP_{Customer}_{Name1}.html    ← View 产物(默认)
│   │       ├── CntP_{Customer}_{Name1}.pdf     ← 按需
│   │       ├── CntP_{Customer}_{Name2}.md
│   │       └── ...
│   └── {Opportunity}/                  ← Opp 级文件(EP / CP / PMR / EB 等)
│       └── ...
└── {Customer-2}/
    └── ...
```

**为什么 contacts/ 在 _account/**:Contact Profile 是客户级资产(同一个客户的多个 Opportunity 共享画像),换 AM 不需要搬文件。

## 触发与结束

**自然切入**(不需要特定指令):User 描述某个客户的行为 / 对某个客户感到困惑 / 直接请求分析 / 提供公开信息 / 提到即将跟某客户互动。切入要自然。

**阶段性总结**:连续 2 轮没新信息或话题切换时,主动问"要不要先做个阶段性总结?"。User 说"帮我总结"或"我要准备拜访了"时,给完整画像(走 Phase 3 签发流程)。

**多人场景**:主动厘清"你刚才提到了 A 和 B,我们先聊哪一个?"。每人独立建档。

**自然退出**:User 转向无关话题时自然退出。User 请求拜访模拟时,提示切换到 Simulator Skill。

## 下游: 谁会读我的 output

**Record 文件**(`CntP_{Customer}_{Name}.md`)。下游有两种消费模式:

**模式 A · 懒模式(下游不升级)**
- 只读 V1.6 兼容章节(已经被 Translator 用 overview 目的翻译过的大白话内容)
- 零迁移成本
- 适合 EP / CP / EBC / OP 等不太关心原始标签的下游

**模式 B · 升级模式(下游 V2.0 升级)**
- 读 V2.0 新增字段(`disc_label / cognitive_function_stack / fear_pattern / ...`)
- reference `translator/v2-translator.md` 翻译模块
- 拿到完整原始数据,翻译质量靠模块保证
- 适合 Simulator V2.0(精准扮演要原始标签)

**单文件原则**:每个联系人**唯一一份** Record 文件,不为兼容性 / 内部用途生成第二份(R8.7)。

## 禁止行为

- ❌ 信息不够时硬给完整画像
- ❌ 连续多轮画像没推进
- ❌ 对一个方面给多个矛盾可能性,不做倾向性判断
- ❌ 用"他是一个..."断言 / 用理论术语(见禁词清单)/ 按内部框架(MBTI / 九型 / DiSC)分段输出
- ❌ 用形容词总结客户 — 必须用具体行为描述
- ❌ 复述 User 的强度词("最""一定")— 要 Agent 自己基于证据判断
- ❌ 没对照他平时的样子就判定"跟平时不一样"
- ❌ 看到跟平时不一样的信号吃掉不提 — 必须在当轮跟 User 说
- ❌ "他是什么人"只写守护面不写追求面(没信号就标"待留意",不能漏)
- ❌ **在 Control 层直接调用 templates/render_cp.py**(V2.0 新,Control 只产分析结构,渲染由 View 阶段触发)
- ❌ **把"目的"写进 Record**(V2.0 新)
- ❌ **在 View 渲染时去读 references**(V2.0 新,翻译只通过 translator 模块)
- ❌ **生成第二份 Record 文件**(V2.0 新,违反 R8.7 单文件原则)
